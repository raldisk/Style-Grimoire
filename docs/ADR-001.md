# ADR-001 — Deployment Architecture & API Interaction Strategy

**Status:** Accepted  
**Date:** 2026-03-08

---

## Summary

This ADR defines the chosen architecture for how Style Grimoire interacts with the Anthropic Messages API and documents a migration path for production deployment. It clarifies the current runtime assumptions, trade-offs, and a prioritized set of enhancements required for robust operation outside the artifact runtime.

---

## Context

Style Grimoire is currently developed and run inside Anthropic's artifact-style runtime environment. The runtime provides:

- Managed authentication and secret injection
- Platform-managed outbound networking
- An isolated sandbox suitable for artifact-style apps

Because of these runtime guarantees, the current client can call the Messages API directly without exposing static client secrets. This arrangement is runtime-specific — deploying outside this environment requires an alternate architecture.

---

## Decision

**1. Current (accepted):** Keep client → Anthropic direct calls while running inside the artifact runtime. This is the simplest approach and fits the current development and demo workflow.

`import.meta.env.PROD === false` → `API_URL = "https://api.anthropic.com/v1/messages"`

**2. Production (implemented):** A backend Express proxy handles all API calls when deployed standalone. The API key lives exclusively on the server — never in the browser bundle.

`import.meta.env.PROD === true` → `API_URL = "/api/generate"` → `server/index.js` → Anthropic

```
React UI → /api/generate → server/index.js → Anthropic Messages API
```

---

## Rationale

The artifact runtime simplifies local development and demonstration. It removes the immediate need for a server and accelerates iteration. For public or production deployments, the backend proxy architecture is required because:

- It prevents API key exposure in browsers
- It centralizes retry/backoff and rate-limiting strategies to reduce cost and improve resilience
- It enables observability (central logs, metrics) and authorization models (per-user quotas)
- It provides a place to implement PII redaction, request validation, and abuse prevention

---

## Consequences

**Positive:**
- Fast development cycles and simple demos in the artifact runtime
- Clear, implemented migration path to production
- API key never in browser bundle in either context

**Negative / caveats:**
- Client code must be hardened even in the artifact runtime (error handling, history trimming) to avoid crashes and UX regressions
- A server introduces operational cost and complexity

---

## Security Considerations

| Concern | Mitigation |
|---------|-----------|
| API key exposure | Server-side env var only — never in browser or Docker image |
| Denial-of-wallet attacks | Server-side rate limiting per IP, usage caps (future) |
| Denial-of-service via streaming | Enforced request timeout + AbortController cancellation |
| Malicious input / prompt injection | Validate and sanitize user inputs; never eval model outputs |
| Log leakage | Do not store prompts/responses in plaintext logs in production |
| CSRF / open redirect | Restrict CORS to trusted origins when proxy is public-facing |

---

## Implemented Hardening (Applied)

All of the following are implemented in the current codebase:

- ✅ `res.ok` check before parsing JSON response body
- ✅ `try/catch` around `res.json()` with meaningful fallback error
- ✅ `crypto.randomUUID()` for collision-resistant message IDs
- ✅ Functional `setHistory(prev => ...)` updates — safe under React concurrent rendering
- ✅ `MAX_HISTORY = 10` turn cap — prevents token overflow in long sessions
- ✅ `403` handling in `classifyError`
- ✅ `AbortController` — cancels in-flight request on new submission or style switch
- ✅ `retryFetch` — exponential backoff with jitter (1s+jitter → 2s+jitter → 4s+jitter) on 429/529
- ✅ Express proxy (`server/index.js`) — injects `x-api-key` + `anthropic-version` server-side
- ✅ Turn count via `filter(m => m.role === "user").length` — immune to system message additions

## Recommended Next Steps (Non-Blocking)

These are improvements for when the project scales beyond a personal workbench:

**Short-term (if moving to public deployment):**
- Add Zod runtime schema validation for request payloads and streaming events
- Add per-IP rate limiting middleware in `server/index.js`
- Add unit tests for `classifyError`, history trimming, and `retryFetch` logic
- Add basic telemetry: request latency, error counts, 429 rate, avg tokens per request

**Medium-term:**
- SSE token streaming — reduces perceived latency by 2–4s on longer responses
- Persistent message store with optimistic UI updates
- PII redaction in production logs
- Auth layer + per-user usage quotas on the proxy

---

## Migration Path (High-Level)

If the project evolves into a fully public production application:

1. ✅ Implement client hardening (res.ok, safe parse, trimming, functional sets, retryFetch)
2. ✅ Add server proxy (`server/index.js`) with secret management
3. Add retry/backoff logic server-side (currently client-side via `retryFetch`)
4. Add authentication and per-user quotas on the proxy
5. Roll out observability: request metrics, error rates, latencies, token usage
6. Migrate traffic incrementally and monitor
7. Add SSE streaming passthrough when ready
8. Complete cutover and deprecate direct client → API path

---

## Operational Checklist (Before Public Exposure)

- [ ] Backend proxy implemented and `ANTHROPIC_API_KEY` stored in secure vault / env
- [ ] Zod validation exercised with unit tests
- [ ] Rate limiting, quotas, and retry policies implemented server-side
- [ ] Telemetry and alerting configured (error and latency SLOs)
- [ ] Regression tests for streaming parsing and cancellation (if SSE added)
- [ ] Data retention and log retention policies documented
- [ ] CORS restricted to trusted origins
- [ ] Docker image scanned for vulnerabilities (Trivy or equivalent)

## Running Self-Hosted

**Docker (recommended):**
```bash
docker build -t style-grimoire .
docker run -p 3000:3000 -e ANTHROPIC_API_KEY=sk-ant-... style-grimoire
```

**Local dev (two terminals):**
```bash
# Terminal 1 — Express proxy on port 3000
node server/index.js

# Terminal 2 — Vite dev server on port 5173 (proxies /api → 3000)
npm run dev
```
